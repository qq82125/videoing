# IVD Video Workflow

一个面向 `IVD` 行业内容生产的本地视频工作台。

它把“主题生成 -> 脚本确认 -> 口播/封面试产 -> 文案修订 -> 镜头精修 -> 导出检查”串成一条可反复回看的生产链，适合本地快速打磨竖屏短视频草稿。

## 项目定位

这不是通用剪辑器，也不是单纯的文案生成器。

当前项目更像一个**本地内容生产工作流**：

- 先确定选题、角度、受众和时长
- 再生成可修改的口播脚本
- 再按成本逐步生成口播、字幕、封面和镜头
- 最后做导出前检查与成片导出

整个过程以 `draft` 为核心资产，所有中间结果都会落到本地草稿目录，方便继续生产、回看和修改。

## 当前真实工作流

前台当前按 5 步组织：

1. `预制阶段`
   - 输入主题或标题/正文
   - 生成首版脚本、标题方案、封面方向
   - 先确认口播结构和分段节奏

2. `开始生产并复核结果`
   - 先生成 `口播 / 字幕`
   - 再生成 `封面`
   - 再按需补 `当前镜头素材`
   - 同一区域里查看预览、字幕切分、封面和导出前检查

3. `需要时先改文案`
   - 修改标题、Hook、正文、CTA
   - 修改封面标题、副标题、视觉提示词和高级排版
   - 改完后按影响面分别同步：
     - `正文改完：同步口播 / 字幕`
     - `标题封面改完：同步封面`

4. `最后进入镜头精修工作台`
   - 查看 `scene` 列表
   - 逐镜查看当前素材
   - 调整镜头标题、类型、时长、口播
   - 按需局部重做当前镜头

5. `准备出片前再检查`
   - 执行导出前质检
   - 查看是否存在阻塞项
   - 生成导出脚本，若环境具备 ffmpeg 则直接导出视频

一句话理解：

- 第二步负责“先把产物跑出来”
- 第三步负责“改文案和封面参数”
- 第四步负责“逐镜精修”
- 第五步负责“能不能出片”

## 当前能做什么

- 输入 `主题` 或 `标题 / 正文` 创建草稿
- 生成并切换标题方案、封面风格组合
- 生成 `口播 / 字幕`
- 生成 `封面`
- 生成并重做单个镜头素材
- 在页面内直接修改：
  - 标题
  - Hook / 正文 / CTA
  - 封面标题 / 副标题
  - 封面视觉提示词 / 负向提示词
  - 封面排版参数
  - 镜头标题 / 类型 / 时长 / 口播
- 保留草稿历史并继续生产
- 执行导出前检查
- 生成导出脚本并在本机/容器内直接出片

## 封面能力现状

当前封面已经不是“模型直接出带字成品”，而是拆成了两层：

1. `纯背景图`
   - 可重抽
   - 可保留多个背景版本
   - 可切换当前背景

2. `系统硬叠字`
   - 标题 / 副标题
   - 位置 / 对齐 / 字号
   - 宽度 / 字间距
   - 上下偏移 / 左右偏移

页面里会先显示实时叠字预览，确认后再同步为正式封面。

## 镜头能力现状

当前镜头逻辑是：

- 后端先按脚本构造固定的首轮 `storyboard` 骨架
- 当前默认会形成 6 个首轮 scene
- 前台镜头按钮是**逐镜补素材**，不是“一键全量生成全部镜头”
- 每补完当前镜头，页面会自动切到下一个待补镜头
- 全部镜头补齐后，再进入第四步做逐镜精修

默认的 6 个 scene 大致是：

1. 开场判断
2. 行业现象
3. 变量拆解 A
4. 变量拆解 B
5. 观点判断
6. 结尾引导

## 项目结构

```text
public/                 前端页面、状态管理与模块化面板
public/app/             前端拆分模块（编辑状态、内容面板等）
public/shared/          前后端共享前端辅助模块
src/server.js           服务入口
src/server/             路由与工作流模块
scripts/                回归与冒烟脚本
data/drafts/            草稿与本地产物（不提交）
data/config/            本地模型配置模板
data/logs/              本地运行日志（不提交）
docs/                   补充文档与素材
prompts/                仓库内提示词模板
skills/                 仓库内协作技能说明
```

## 本地启动

要求：

- Node.js 20+
- `ffmpeg`
  - 没有 ffmpeg 时，仍可生成导出脚本，但不能直接导出视频

启动：

```bash
npm install
npm start
```

默认访问：

```text
http://127.0.0.1:3016
```

## Docker 启动

仓库已经提供单容器运行方式。

首次启动：

```bash
docker compose up -d --build
```

查看日志：

```bash
docker compose logs -f
```

停止：

```bash
docker compose down
```

默认端口：

```text
http://127.0.0.1:3016
```

容器特性：

- Node 20 运行时
- `ffmpeg / ffprobe`
- `healthcheck`
- `data/` 持久化挂载
- 基础资源限制（`2 CPU / 2G RAM`）

## 配置方式

项目把不同生产动作拆成独立调用点，当前主要包括：

- `script`：文案生成
- `storyboard`：镜头规划
- `image`：图片生成
- `tts`：配音生成
- `transcription`：音频转写/对齐
- `moderation`：审核与合规预检

配置来源有两类：

1. `.env.local`
   - 提供默认值和兼容旧配置
   - 参考：[.env.local.example](./.env.local.example)

2. `data/config/llm-config.json`
   - 用于每个调用点独立配置
   - 模板：
     - [data/config/llm-config.example.json](./data/config/llm-config.example.json)
     - [data/config/llm-config.proxy-template.json](./data/config/llm-config.proxy-template.json)

同时，前台 `/admin` 已带管理界面，可直接维护模型路由和运行状态。

## 管理台

`/admin` 当前主要负责：

- 运行状态与环境检查
- LLM 路由配置
- 配置校验与问题提示
- 草稿与资产查看入口

当前已经补了：

- `validation` 可视化
- 本地结构化运行日志
- 更明确的配置健康度反馈

## 数据与产物

每条草稿都会在 `data/drafts/<draft-id>/` 下落本地产物，通常包括：

- `draft.json`
- `voice.mp3`
- `subtitles.srt`
- `cover.svg / cover.png`
- `scenes/scene-*.svg|png`
- `export-video.sh`
- `final-video.mp4`（如果成功导出）

## 回归命令

仓库已经补了一组最小回归命令，建议在改动后至少跑对应项：

```bash
npm run smoke
npm run regression:stage1
npm run regression:legacy
npm run regression:assets
npm run regression:preview-audio
npm run regression:cover-compose
npm run regression:storyboard
```

各自关注点：

- `smoke`
  - 基础入口与接口通路
- `regression:stage1`
  - 生成 -> 保存 -> 口播/字幕 -> 封面 -> 质检 -> 导出
- `regression:legacy`
  - 历史草稿兼容
- `regression:assets`
  - 静态资源与前端拆分模块可访问性
- `regression:preview-audio`
  - 第一阶段口播生成后，前台可看到播放器
- `regression:cover-compose`
  - 封面背景与叠字链路
- `regression:storyboard`
  - storyboard 补素材与第四步工作台联动

## 当前已知边界

目前这套系统已经能闭环生产，但还不是一个完全自由的剪辑器。已知边界包括：

- 镜头首轮仍以固定骨架为主，不是完全自由编排
- 镜头生产当前更偏“逐镜补素材”，不是一键全量整轮生成
- 图片模型仍可能偶发生成伪文字或不够稳定的背景
- 前后端预览与最终栅格化虽然已经尽量拉齐，但字体渲染仍可能有轻微差异
- 本地环境缺少 ffmpeg 时只能生成导出脚本，不能直接出片

## 适合谁

当前版本更适合：

- 需要快速把行业选题变成短视频草稿的人
- 需要频繁修改文案、封面和镜头方向的人
- 希望把“内容生产”和“导出判断”拆开的团队
- 能接受先跑本地工作流、再逐步精修的人

## 不适合什么

当前版本不适合：

- 追求完整 NLE 剪辑体验
- 需要多人在线协同编辑
- 需要复杂时间线、轨道和音视频混剪
- 希望完全自动生成并一键无人工审校出片

## 当前仓库策略

默认不提交这些本地产物和本地配置：

- `node_modules/`
- `data/drafts/`
- `data/logs/`
- `data/config/llm-config.json`
- `.env.local`

## 最近这轮升级的核心变化

相对早期版本，当前仓库已经完成了几件关键收口：

- 后端从单文件入口拆成 `src/server/` 模块
- 前端从大单文件逐步拆成 `public/app/` 与 `public/shared/`
- 增加了阶段性回归脚本
- 封面工作流改成“纯背景 + 硬叠字”
- 第三步文案区改成更贴合主逻辑的修订工作台
- 镜头区补了“逐镜补素材并自动推进”的流程

## 推荐使用顺序

如果你第一次上手，建议按这个顺序使用：

1. 生成草稿
2. 确认脚本并进入生产
3. 先生成口播 / 字幕
4. 再生成封面
5. 不满意就去第三步改文案
6. 再回来同步对应产物
7. 之后再补镜头并进入第四步逐镜精修
8. 最后做导出前检查和导出

这也是当前产品最顺的主路径。
